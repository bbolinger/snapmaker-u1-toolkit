# Snapmaker U1 + OrcaSlicer + Moonraker Notes

Session-derived notes for the operator's Snapmaker U1 workflow. Treat as a reference pattern, not universal truth for every U1.

## Proven safe envelope

Proven in-session:

```text
STL → OrcaSlicer headless → U1 PETG G-code → Moonraker upload with print=false → printer remains idle/cool
```

Do **not** treat this as approval for unattended print start.

## Printer/control facts observed

- Printer reachable through Moonraker-style API on port `7125`.
- `gcodes` root was writable and mapped to `/userdata/gcodes`.
- Upload endpoint accepted multipart upload at `/server/files/upload` with `print=false`.
- Upload response included `print_started: false` and `print_queued: false` for no-start dry run.
- Printer state after upload stayed inactive/cool.

## OrcaSlicer headless notes

OrcaSlicer AppImage can be extracted and run headless without FUSE by using `--appimage-extract` and invoking `squashfs-root/bin/orca-slicer`.

In this environment, missing GUI/runtime libraries were satisfied by locally downloading/extracting Debian packages and prepending their lib directory to `LD_LIBRARY_PATH`. Do not record “Orca does not work” if the direct AppImage fails; the fix is extraction + library path.

Useful command shape:

```bash
LD_LIBRARY_PATH="<orcaslicer-install>/local-libs/usr/lib/x86_64-linux-gnu:<orcaslicer-install>/squashfs-root/usr/lib:<orcaslicer-install>/squashfs-root/usr/lib/x86_64-linux-gnu:$LD_LIBRARY_PATH" \
  <orcaslicer-install>/squashfs-root/bin/orca-slicer \
  --load-settings "<machine.json>;<process.json>" \
  --load-filaments "<filament.json>" \
  --outputdir <outdir> \
  --slice 0 \
  <model.stl>
```

## Profile-loading pitfall

Bundled Snapmaker profiles exist in OrcaSlicer 2.4.0, including:

```text
resources/profiles/Snapmaker/machine/Snapmaker U1 (0.4 nozzle).json
resources/profiles/Snapmaker/process/0.16 Optimal @Snapmaker U1 (0.4 nozzle).json
resources/profiles/Snapmaker/filament/Snapmaker PETG @U1.json
```

But headless profile inheritance/filament loading produced misleading output in testing:

- `filament_settings_id` looked like PETG, but `filament_type` became `PLA`.
- Nozzle/bed temps defaulted to PLA-like values in some runs.
- A child process profile without flattened inheritance produced `layer_height = 0.2` despite the intended `0.16` preset.

Fix pattern:

- Flatten/merge the profile inheritance chain into explicit local profiles.
- Override concrete values derived from known-good G-code, especially material, nozzle temp, bed temp, bed type, layer height, infill, and printer model.
- Validate generated G-code metadata after slicing.

## Working local profile pattern from session

A custom PETG filament profile was created by merging:

```text
filament/fdm_filament_common.json
filament/fdm_filament_petg.json
filament/Snapmaker PETG @U1 base.json
filament/Snapmaker PETG @U1.json
```

Then explicitly setting:

```json
{
  "filament_type": ["PETG"],
  "filament_density": ["1.25"],
  "filament_diameter": ["1.75"],
  "filament_flow_ratio": ["0.95"],
  "nozzle_temperature": ["255"],
  "nozzle_temperature_initial_layer": ["255"],
  "nozzle_temperature_range_low": ["230"],
  "nozzle_temperature_range_high": ["270"],
  "hot_plate_temp": ["80"],
  "hot_plate_temp_initial_layer": ["80"],
  "textured_plate_temp": ["80"],
  "textured_plate_temp_initial_layer": ["80"],
  "bed_type": ["Textured PEI Plate"],
  "compatible_printers": ["Snapmaker U1 (0.4 nozzle)"]
}
```

A custom process profile was created by flattening/merging:

```text
process/fdm_process_U1.json
process/fdm_process_U1_common.json
process/fdm_process_U1_0.16.json
process/0.16 Optimal @Snapmaker U1 (0.4 nozzle).json
```

Then explicitly setting:

```json
{
  "curr_bed_type": "Textured PEI Plate",
  "print_settings_id": "Hermes 0.16 Optimal @Snapmaker U1 Textured PEI"
}
```

Generated G-code from the flattened profiles validated as:

```text
; printer_settings_id = Snapmaker U1 (0.4 nozzle)
; print_settings_id = Hermes 0.16 Optimal @Snapmaker U1 Textured PEI
; filament_type = PETG
; layer_height = 0.16
; nozzle_temperature = 255
; nozzle_temperature_initial_layer = 255
; first_layer_bed_temperature = 80
; first_layer_temperature = 255
; curr_bed_type = Textured PEI Plate
; sparse_infill_density = 15%
; wall_loops = 2
```

## Moonraker no-start upload pattern

Upload G-code with multipart form data and `print=false`. Verify both the upload response and printer state afterward.

Expected successful dry-run indicators:

```json
{
  "action": "create_file",
  "item": {"root": "gcodes", "path": "...gcode", "permissions": "rw"},
  "print_started": false,
  "print_queued": false
}
```

Then query printer objects and confirm:

```text
virtual_sdcard.is_active = false
heater_bed.target = 0
extruder.target = 0
print_stats.state is not printing
```

## Camera/preflight relation

The U1 camera can be triggered via Moonraker websocket `camera.start_monitor`, then `server/files/camera/monitor.jpg` can be fetched. That is useful for preflight, but it is not the main slicing workflow and may not prove bed-clear due to camera angle. Fail closed if bed visibility is incomplete.

## Recommended operator answer after dry-run upload

Report concise evidence:

- source STL
- slicer version
- profiles used
- generated G-code path
- key metadata values
- upload response `print_started=false`
- printer state after upload
- remaining blocker / approval required

Never imply the job is safe to start solely because slicing/upload succeeded.
