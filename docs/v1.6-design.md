# v1.6 (A) Design — Pre-Slice Orca Analysis Pass

Captures the design decisions and empirical evidence behind the v1.6 (A)
pre-slice analysis feature. The goal: give the operator Orca's *real*
mesh-topology overhang verdict during the analysis phase, BEFORE they commit
to a slice — replacing today's conservative-stupid face-angle approximation.

## Problem this solves

The v1.5.2 workflow renders both orientations and labels them with a
face-angle metric (`overhang_area_pct` + `supports_tier`). That metric
flags any face above an angle threshold, ignoring:

- **Span length** — a 60° face spanning 5mm prints fine; spanning 60mm sags
- **Curvature continuity** — a sphere's hemisphere self-supports because
  each layer extends only fractions of a mm past the previous
- **Per-layer offset** — what Orca's real algorithm actually looks at
- **Bridge length** — horizontal unsupported spans have their own thresholds

Result: face-angle systematically overestimates risk. Operator-observed gap
on `wall_mount_laid_on_back`:

```
face-angle metric: 39.1% / "very heavy"
Orca real verdict: 8.8% of layers tagged overhang
```

4× more conservative than reality. Operator's eyes saw the difference
during a live test; the metric didn't.

## Design

### Signal architecture

Two distinct signals from Orca, two different layers of analysis:

| Signal | Source | What it tells us | Invariant to |
|---|---|---|---|
| `warning_message` | Orca **mesh-topology analysis** | "floating cantilever" / "floating regions" — geometric red flags | Wall count, infill density, top/bottom layer count, layer height |
| `;TYPE:Overhang wall` count | Orca **per-layer print-path analysis** | What % of layers have any tagged overhang segments | Wall count, infill density. **NOT** layer height. |

The mesh-topology warning is the cleanest signal — it's about the shape, not
the print settings. The per-layer count is a secondary numeric that depends
on slice geometry.

### Workflow integration

Analysis phase gains a **draft slice** step per orientation:

1. Materialize a draft process profile = production profile + overrides:
   - `wall_loops = 1`              (skip extra perimeter compute)
   - `sparse_infill_density = 0%`  (skip infill paths)
   - `top_shell_layers = 0`        (skip top solid)
   - `bottom_shell_layers = 0`     (skip bottom solid)
   - `gcode_thumbnails = 0`        (skip thumbnail encoding)
   - `enable_support = 0`          (analysis isn't choosing supports)
2. Keep production `layer_height` unchanged (critical — see below).
3. Run Orca with `--mstpp 30` as a safety cap.
4. Parse `result.json.sliced_plates[0].warning_message` → categorize.
5. Parse gcode `;TYPE:Overhang wall` tags → compute layer-fraction.
6. Surface verdict in the orient need_input event via the `note` field.

The agent's job is unchanged — surface the workflow's pre-computed verdict
verbatim, tool-call the matched option's `next_command`. No LLM-side
interpretation of risk levels or numeric thresholds.

### Warning categorization (in code)

```python
def categorize(warning_text: str) -> str:
    txt = (warning_text or "").lower()
    if not txt or txt == "null":         return "CLEAN"
    if "floating cantilever" in txt:     return "CANTILEVER"
    if "floating region" in txt:         return "FLOATING_REGIONS"
    if "overhang" in txt:                return "OVERHANG_FLAGGED"
    return "UNKNOWN"
```

These categories drive the `note` text the operator sees and the
`recommended` flag on the orient options.

### Clean-vs-risky threshold (in code)

```python
def is_clean(category: str, overhang_pct: float) -> bool:
    return category == "CLEAN" and overhang_pct < 10.0
```

If as-authored is clean → skip the auto-orient draft slice (operator
usually wants as-authored anyway). Operator override always available by
picking the auto option, which re-runs analysis with `--orient auto`.

## Empirical validation

Five fixtures × five draft-profile variants × three measurements
(wall-clock, warning_category, overhang_pct) on the Hermes-host
OrcaSlicer binary via the docker-exec test rig. Production process
profile = `0.20 Strength @Snapmaker U1 (0.4 nozzle)` with 6 walls, 25%
infill, 4 top-bottom layers. Filament = `Snapmaker PETG @U1`. Machine
profile = stock Snapmaker U1 0.4-nozzle.

Reproducible rig: [`tests/experiments/v16-draft-validation.sh`](../tests/experiments/v16-draft-validation.sh).
Raw results gitignored (live in `/opt/data/snapmaker_u1/v16-experiment2/results.tsv`
on the operator's host).

### Results (4 valid fixtures, overhang_fan excluded)

#### Speed (sum of wall-clock across the 4 fixtures)

```
ground: 45.5s   (baseline — full production slice)
v1:     17.1s   (2.7× speedup)
v2:     23.0s   (2.0× speedup)
v4:     23.4s
v5:     26.1s
```

Largest single fixture (`wall_mount_auto`, 765 layers at production 0.20mm)
takes 16s on v2. Well under the 30s `--mstpp` cap.

#### Warning categorization — PERFECT on all variants

| Fixture | Ground truth | v1 | v2 | v4 | v5 |
|---|---|---|---|---|---|
| `wall_mount_source` | CANTILEVER | ✓ | ✓ | ✓ | ✓ |
| `wall_mount_auto` | CLEAN | ✓ | ✓ | ✓ | ✓ |
| `caulkholder_source` | FLOATING_REGIONS | ✓ | ✓ | ✓ | ✓ |
| `benchy` | CLEAN | ✓ | ✓ | ✓ | ✓ |

**This is the proof that warning detection is mesh-topology, not print-settings dependent.** All four variants gave the same Orca verdict on every fixture, even with wildly different wall/infill/top-bottom settings.

#### Overhang % delta from ground

| Fixture (ground %) | v1 | v2 | v4 | v5 |
|---|---|---|---|---|
| caulkholder (27.7%) | +28.6pp | **+0.0pp** | +28.6pp | +28.6pp |
| wall_mount_auto (4.6%) | +2.9pp | **−0.2pp** | +2.9pp | +2.9pp |
| benchy (14.2%) | +8.9pp | **−0.4pp** | +8.9pp | +8.9pp |
| wall_mount_source (8.8%) | +13.3pp | **+0.0pp** | +13.3pp | +13.3pp |
| **avg delta** | **+13.4pp** | **−0.1pp** | **+13.4pp** | **+13.4pp** |

**v2 is the only variant whose `overhang_pct` directly predicts the commit
slice's number.** v1/v4/v5 all over-count because they use 0.30mm layer
height — each draft layer represents 1.5× the vertical distance, so a fixed
physical overhang region gets flagged in proportionally more layers.

### Why v2 wins

- **Warning category match: perfect** (true of all variants — confirms mesh-topology robustness)
- **Overhang % match: ±0.4pp on every fixture** (only variant where this holds)
- **Speed: 2× faster than ground** (45s → 23s on the 4 fixtures combined)
- **Layer-height anchor: production 0.20mm** — the draft's numbers translate 1:1 to what the operator's real print will show

### Why we drop walls / infill / top-bottom

NOT because the warning lives there. Because computing them takes Orca
compute time without changing what we measure:

- 6 walls vs 1 wall = Orca generates 6× perimeter paths per layer
- 25% infill vs 0% = Orca generates infill patterns
- Top/bottom solid layers = Orca generates dense fill on those layers

None of those reach the mesh-topology code path or the overhang detection
code path. We're not lowering Orca's overhang sensitivity. We're telling
Orca: skip the print-quality work we'll never look at.

### Why we keep production layer_height

This is the load-bearing decision. The operator's real print runs at
0.20mm. When the draft matches, every `;TYPE:Overhang wall` tag in the
draft corresponds 1:1 to a tag in the eventual commit slice. The
percentages are commit-predictive. v1's 0.30mm matched the warning
category (mesh-topology is layer-invariant) but inflated the overhang_pct
by +14pp average because each draft layer represents more vertical
distance.

## Real-world grounding (operator's note)

- 3DBenchy is the print Snapmaker ships pre-built in the U1's onboard
  memory. The operator did not modify it. Our v2 verdict on 3DBenchy is a
  match against the manufacturer's own reference print.
- `wall_mount_laid_on_back` and `CaulkHolderWallMount` are real
  engineering parts the operator prints for load-bearing applications,
  not torture-test toys.
- The community-shared "0.20 Strength" profile is essentially Snapmaker
  stock + a filament-brand temperature tweak (off-brand PETG). It is not
  a heavily customized one-off.
- All test data comes from the operator's actual U1 environment, not a
  sandboxed simulation.

## Adjacent investigations that informed this design

### `--export-slicedata` / `--load-slicedata` cache reuse — NEGATIVE

Spike test: produce slicedata from the draft slice, feed it to the commit
slice via `--load-slicedata`. Hypothesis: commit slice gets faster
because Orca skips re-computing geometry.

Result: **5.3s baseline vs 6.0s cached (642ms slower).** The 20MB
slicedata blob's read overhead outweighs any saved compute, and the cache
from a 0.30mm draft can't be reused for the 0.20mm commit anyway
(different layer geometry).

**Conclusion: don't pursue caching.** The draft cost is real (+2–16s
depending on fixture size) and not recoverable.

### Polygon-count / STL size cap

OrcaSlicer can slow down on 1M+ triangle models. For v1.6:

- Hard cap via `--mstpp 30` (per-plate slice time limit)
- Don't decimate the mesh: changes the geometry the operator sees in the
  render, and community advice is "usually unnecessary"
- For huge STLs, the draft slice naturally takes longer — accept that or
  hit the cap

## Implementation plan (next session)

### Workflow changes

1. New helper `_draft_slice_analysis(stl, out_dir, machine, process, filament)`:
   - Materialize draft profile from production process profile + v2 overrides
   - Invoke Orca with `--mstpp 30`, timeout 60s subprocess-level
   - Parse `result.json` + gcode
   - Return `{warning_message, warning_category, overhang_pct, layer_count, draft_time_ms}`
2. New helper `_categorize_warning(text) -> str` (CLEAN / CANTILEVER /
   FLOATING_REGIONS / OVERHANG_FLAGGED / UNKNOWN)
3. New helper `_is_clean(category, overhang_pct) -> bool` (CLEAN AND < 10%)
4. Analysis-phase logic:
   - Render + draft-slice as-authored
   - If as-authored clean → skip auto-orient draft (label "not analyzed")
   - Else → render + draft-slice auto-orient
   - Decide recommendation based on Orca verdicts
5. Orient need_input event payload:
   - Same shape as v1.5.2 (two options always: as-authored / auto-orient)
   - `note` field carries the Orca verdict in plain language
   - `recommended` flag follows Orca's verdict
   - New audit-only field: `orient_risk: {as_authored: {...}, auto: {...}}`
   (agent doesn't read this; it exists for testing + observability)
6. Graceful failure: if draft slice rc != 0, fall back to today's
   face-angle metric in the note, mark `orient_risk.as_authored = "draft_failed"`

### Skill changes

None. The agent's job is identical: surface `prompt` + options + `note`

## What actually shipped — HERMES.md and the stable-tier insight

While shipping v1.6 (A) the harness caught a regression: with the longer
`note` text in the orient prompt, the agent's collision-resolve tool call
on turn 7 was being dropped, deterministically. The skill body itself
was correct; the agent was just *forgetting* the procedural rule by the
time it reached the collision step.

Research into Hermes' prompt-assembly architecture surfaced the
three-tier layout: **stable** (identity, tools, skills index),
**context** (project files: `.hermes.md` / `HERMES.md` / `AGENTS.md`),
**volatile** (memory snapshot, conversation turns). Skills load into
turns, which means the full skill body lives in the *volatile* tier and
is subject to compression once context pressure crosses the threshold.
`.hermes.md` lives in the *stable+context* tier and survives compression.

This isn't unique to Hermes. The same pattern shows up in other agent
runtimes (an agent's rules file + persistent memory, Cursor's
`.cursorrules`, etc.) — **a stable scaffold for procedural rules that
must remain present regardless of conversation length**. It is not a
small-model crutch; large agents need it for the same reason, and the
toolkit author has himself been caught dropping rules from his own
stable-tier memory while developing this very file.

The toolkit's v1.6 (A) solution:

- Workspace `HERMES.md` at the repo root carries the four most-critical
  rules. They are inviolable — surface them on every turn regardless of
  conversation history.
- `deploy_to_runtime.sh` writes a persistent copy at
  `/opt/data/.hermes.md` (= host `/appdata/hermes/.hermes.md` via the
  bind mount) and uses `docker exec` to also place a copy at
  `/opt/.hermes.md` inside the Hermes container, which is where
  Hermes' `prompt_builder._find_hermes_md` finds it while walking up
  from `/opt/hermes` (the daemon's CWD).
- `SKILL.md` keeps the full procedural runbook unchanged.

Empirical evidence:

| Configuration | Harness runs | Criteria pass |
|---|---|---|
| v1.6 implementation only | 2 | 3/5 (criteria 3+4 dropped on collision-resolve) |
| v1.6 + HERMES.md (stable-tier rules) | 1 | 5/5 (turn 7 events=9, gemma chained collision-resolve + Stage 1) |

The pattern is straightforward: anything the agent MUST do regardless
of conversation length goes in HERMES.md. Anything the agent should do
when relevant goes in SKILL.md.

## Acceptance criteria (v1.6 (A))

Adding to the design contract:

> **9. Orient prompt carries Orca's verdict.** The orient need_input event
> emitted in the analysis phase has a `note` field that surfaces Orca's
> categorized warning (CLEAN / CANTILEVER / FLOATING_REGIONS / OVERHANG_FLAGGED)
> and a layer-fraction overhang estimate, both derived from a real draft slice
> of the source STL, not from face-angle approximation.

> **10. Clean-case skip.** When as-authored is categorized CLEAN with
> overhang_pct < 10%, the workflow does NOT run the auto-orient draft slice
> (operator override path still available via picking the auto option).

> **11. Draft slice failure is non-fatal.** If the draft slice returns rc !=
> 0, the workflow falls back to the v1.5.x face-angle metric in the note,
> marks `orient_risk = "draft_failed"`, and proceeds with the rest of the
> analysis phase unchanged. The operator sees a clearly-marked fallback,
> never silent fabrication.
