# v2.1.0 — Multi-part / Multi-plate STL Kit Support — Implementation Plan

**Status:** Planned, not started. No code written yet.
**Branch:** `v2.1.0-dev`
**Date:** 2026-06-28
**Predecessor:** v2.0.1 (Safe AI Print Operator + doc patch), on `main`.

This doc is the durable plan record so the work survives context loss. It captures
the *evidence* the design rests on (the spike results), the dispatch model, the data
model, the phase breakdown, and the one open design decision (how decisions are
collected from the operator). Read this first before touching code.

---

## 1. Motivation

Most Printables/MakerWorld models ship as **multiple parts** meant to be printed
together — a person drops N STLs onto one bed, auto-arranges (and sometimes
auto-rotates) them, and prints the whole kit. If they don't all fit, the parts
spill onto multiple plates printed over time.

Today the toolkit treats **one STL → one gcode → one print**. A zip is reduced to
its *first* STL (`_extract_from_zip` picks `stls[0]`). We want to ingest the whole
kit, arrange it, slice it into however many plates it needs, and run it through the
existing safety workflow.

## 2. Spike findings — the evidence this plan rests on

Orca only executes inside the **`hermes-agent-stack`** container (this/Alpine host
can't run it — FUSE/glibc). Spikes were run there via
`docker exec hermes-agent-stack env PYTHONPATH=/opt/data/scripts /opt/hermes/.venv/bin/python3`.
Interpreter note: system `python3` in that container lacks numpy; the toolkit deps
live in `/opt/hermes/.venv`.

Profiles used (real vendor profiles inside Orca's own tree, so `inherits` resolve):
- machine: `…/resources/profiles/Snapmaker/machine/Snapmaker U1 (0.4 nozzle).json`
- process: `…/resources/profiles/Snapmaker/process/0.20 Standard @Snapmaker U1 (0.4 nozzle).json`
- filament: resolved + flattened via the toolkit's own `filament_path('PLA','0.4')` + `_materialize_flat_filament`

### Results

| Test | Input | Flags | Result |
|------|-------|-------|--------|
| A | 4 small cube STLs | `--arrange 1 --slice 0` | `rc=0` → **`plate_1.gcode`** (all fit one bed) |
| B | 9× 90 mm cube STLs | `--arrange 1 --slice 0` | `rc=0` → **`plate_1.gcode`, `plate_2.gcode`, `plate_3.gcode`** (overflow auto-split) |
| C | 4 small cube STLs | `--arrange 0 --slice 0` | `rc=206`, "run found error, return -50" → **fails** |
| D | 3× **real** EGO String Trimmer holder STL | `--arrange 1 --slice 0` | `rc=0` → **`plate_1.gcode`** (all fit) |

### Conclusions (load-bearing)

1. **Multi-STL auto-arrange works headless** via *positional STL args* + `--arrange 1`.
   No segfault. (The upstream CLI crash — OrcaSlicer #11597 — is the
   `--load-assemble-list` path specifically. **We never use `--load-assemble-list`.**)
2. **Orca auto-splits overflow into `plate_N.gcode`** by itself. We need **no
   bin-packer and no 3MF writer.**
3. **`--arrange 1` is mandatory for kits.** With `--arrange 0`, multiple objects
   collide and the slice fails (rc=206). Always pass `--arrange 1` when >1 object.
4. **Output is raw `plate_N.gcode`** in `--outputdir` (not `.gcode.3mf`) with plain
   `--slice 0`. The existing output scanner must collect *all* `plate_*.gcode`,
   not just the newest one.
5. **Relevant CLI options confirmed present** (`orca-slicer --help`):
   - `--arrange 0|1|other` — 0 disable, 1 enable, other auto
   - `--allow-rotations` — arrange may rotate objects in-plane to pack tighter
   - `--orient 0|1|other` — best-face-down orientation (separate from arrange)
   - `--rotate`, `--rotate-x`, `--rotate-y` — manual per-axis rotation
   - `--allow-multicolor-oneplate` — (relevant to the deferred per-part color branch)
   - `--assemble` — merges models into ONE model; **not** what we want (we want
     separately-arranged objects), noted to avoid confusion
   - `--slice 0` = all plates, `--slice i` = plate i
6. **`--orient 1` + `--arrange 1` + `--allow-rotations` compose in one invocation** —
   so the "auto-rotate choice" the operator makes feeds flags into a single slice call.

Spike scripts are session scratch only (`/tmp/.../scratchpad/arrange_spike*.py`) and
were run against container `/tmp`; nothing was written to the operator data dir.

## 3. Dispatch model (operator design)

> Slice all selected parts in one Orca call → Orca emits `plate_1…plate_N.gcode` →
> **upload all N** to the printer → **gate only plate 1's start** through the existing
> v2.0 moat → tell the operator: *"plates 2–N are on the printer; start them from the
> Snapmaker app."*

The Snapmaker app's own auto-arrange is trusted and easy; the toolkit's value is the
**safe operator workflow**, not re-implementing arrange. Gating only plate 1 means the
v2.0 safety core (request_id, audit, `can_start()`, Stage 1/2 token + bed photo) is
reused **almost unchanged** — plates 2–N are honest upload-only records. This is the
key simplification: **no plate-indexed safety rewrite.**

**Decision order (footprint depends on orientation):**
ingest → select parts → orient / auto-rotate choice → arrange + slice (one call) →
upload all plates → gate plate 1.

---

## 4. Data model — `request.json` schema_version 2 (minimal)

Single STL stays a **kit of 1** so all v2.0 behavior and tests are preserved.

- `kit.parts[]` — `{part_id, filename, model_hash, selected}`
- `kit.orient_mode` — `as_authored | auto`  (the auto-rotate choice → drives `--orient` / `--allow-rotations`)
- `plates[]` — upload bookkeeping: `{plate_idx, gcode_path, gcode_hash, printer_storage_filename, uploaded, started}`
- **Top-level `gcode_hash` / `approval` / approval-token stay bound to plate 1.**
  The v2.0 safety core validates plate 1 exactly as today. Plates 2…N are upload-only.
- `migrate_v1_to_v2.py` — wraps an existing request as kit-of-1 with `plates=[plate 1]`.
  Idempotent, never fabricates approval state (same discipline as `migrate_v0_to_v1.py`).

## 5. Phases

- **A — Kit ingest:** `_extract_from_zip` returns *all* STLs; accept multiple file
  args; build `kit.parts[]`. Single-STL path unchanged. Tests + migrator.
- **B — Selection:** decision turn "print all N or pick a subset?" → sets `selected`. Tests.
- **C — Orient / auto-rotate:** decision turn "best-face-down auto-rotate, or keep as
  authored?" → drives `--orient` / `--allow-rotations`. Tests.
- **D — Arrange + multi-plate slice:** ONE invocation, all selected parts,
  `--arrange 1` (+ orient/rotate flags); collect every `plate_*.gcode`; per-plate
  `gcode_hash`; inject thumbnails per plate. Tests.
- **E — Upload-all + readiness card:** upload each plate with a distinct printer
  filename (`<kit>_plate1.gcode`…); readiness card states plate count + "start plate 1
  here, plates 2–N via the Snapmaker app"; top-level binds plate 1. Tests.
- **F — Gate plate 1:** existing Stage 1/2 moat dispatches plate 1 only; `can_start()`
  validates plate 1 (minimal/no change). **All v2.0 single-plate tests must stay green.**
- **G — Skill / docs + release:** SKILL.md kit walkthrough (terse + `references/`),
  one HERMES.md rule, new `docs/events.md` events, CHANGELOG, README "kits" section,
  version bump, migrator doc. **Live-validate on gemma4-26b-64k via Telegram** before
  tagging. Tag **v2.1.0** (additive + migrator = minor; single-STL behavior preserved).

## 6. OPEN DESIGN DECISION — how decisions are collected

Raised by Brent: instead of the LLM walking ~5 sequential decision turns, present a
**single multi-select questionnaire**, collect all answers at once, let the **script**
do the conversion to Orca. "The script is the powerhouse, not the LLM." Still two
triggers (answer the form → slice/readiness; approve photo → start), no session growth.

**Key facts that shape this:**
- The workflow already runs an **ANALYSIS pass first**, so after analysis *all* options
  are known (orient tiers, nozzle-filtered profile list, etc.).
- The script **already accepts every decision in one shot**: passing
  `--orient … --tool … --profile … --supports … --yes` skips the walk and goes
  straight to slice → readiness. The multi-turn Q&A is the *skill's* conversational
  style, **not** a script limitation.
- The one-at-a-time style is **deliberate**: gemma-26b is more reliable answering one
  question at a time. That reliability is the project's north star
  (DESIGN-CONTRACT target = gemma via Telegram).

**Three options on the table:**

1. **Fast-path text questionnaire + keep one-at-a-time fallback (recommended).**
   After analysis, the script can emit one consolidated questionnaire event; the skill
   says: confident operator (or a bigger model like gpt-5.5) answers all at once → LLM
   calls the script once with all flags; gemma unsure → walk one at a time. Cheap
   (mostly skill-doc + one event), keeps the gemma-safe path, multi-select for parts
   drops in naturally. **Does not** shrink the multi-plate plumbing (that's irreducible).

2. **Pure one-at-a-time for v2.1.0**, revisit the questionnaire after kits work.

3. **Native Telegram multi-select UI (poll / Mini App).** Bigger, separate project.
   Pulls the decision out of the LLM/auditable flow, needs extra infra, and gemma
   stops being the thing reasoning about safety. Recommend **defer**.

**Caveats that apply to any choice:**
- The questionnaire only collapses *decision* turns (5→1); the slice/upload/gate
  engine is the same work regardless.
- Small-model multi-answer parsing ("1=auto, parts 1&3&5, T0, 0.20 standard, no
  supports, upload+start") is harder than one question at a time — so a form must be an
  *option*, never the only path.

**DECISION: _pending Brent's pick._**

## 7. Guardrails

- **Lean rule:** every new field/decision must earn its token cost for ONE operator +
  gemma. Reject schema/folder churn that doesn't.
- **Safety moat preserved:** plate 1 gated exactly as v2.0; plates 2–N honest upload-only.
- **Dev discipline:** all work on `v2.1.0-dev`; merge to `main` only after **live
  end-to-end verification** on real hardware (per the dev-branch rule).
- **Ops hygiene:** `--force-with-lease`, per-push permission (never push without
  explicit OK), `chown 10000:10000` after writes to `/appdata/hermes/**`, no scratch in
  the operator data dir (use `mktemp /tmp/`), no token in `.git/config`.
- **No `--load-assemble-list`** (the crashy CLI path). Positional STLs + `--arrange 1`.
